A design library in plain HTML, CSS & JS

One CSS file and one JS file at a public URL style everything I make. No React, no build step, never minified — this page is the library. 

<link rel="stylesheet" href="https://[domain]/lib/facet.css">

Philosophy

Make the web simple again

Modern frontend is a div-soup — minified React machinery you cannot open and understand. Facet bets the other way: plain HTML, CSS and JS, using the browser's own features instead of reaching for a framework. The page you write is the page that ships — readable, commented, and fast because there is no runtime between you and the platform.

The point is to make the internet simpler again by making front-end code good again: web pages built on good practices that just work — quick, easy to maintain, accessible, customizable, and at home on your phone's own OS. This library defines those practices and gives you every one of them for free.

How to use the library

From an empty file to a shipped page in a few steps. Every one is an attribute or a class — there is no build step, nothing to install.

  1. 1

    Add the two tags

    One <link> and one <script> from a public URL — copy them from the top of this page. That is the whole install.

  2. 2

    Start from a layout

    Rename a ready page — landing, app shell, article, pitch deck, document or business card — or assemble one from the block layer. Fill in your words.

  3. 3

    Pick a theme

    One attribute on <html> restyles the whole page — data-theme and data-mode. Or build your own on the Build a theme page and paste the config.

  4. 4

    Assemble with components

    Write semantic HTML with the right classes — buttons, cards, fields, tabs. Every behaviour self-wires from data attributes on load; there is no JS to write.

  5. 5

    Tune the feel, globally

    Density (data-density), type size (data-text-size) and motion (data-motion) are one attribute each — page-wide or per subtree, no per-element work.

  6. 6

    Ship it

    Deploy the static files. A one-line service worker makes the page installable and offline-ready; the head pack and sitemap keep it SEO- and AI-readable.

Pager law. Full-page snap sections that can outgrow

Pager law. Full-page snap sections that can outgrow the viewport are never built with CSS scroll-snap: mandatory re-snaps a tall section back to its top after a flick and wedges the next gesture, proximity feels like no snap, and a free-scrolling outer with a JS settle-snap is mushy and fights momentum. The model that works is the pager .snap ships: the outer never free-scrolls, JS springs its scrollTop between section tops, each section is 100dvh and scrolls natively inside, and gestures hand off at section edges with a rubber-band hint.

Gesture law. Tame iOS with touch-action

Gesture law. Tame iOS with touch-action: pan-y on scrollers (vertical scroll only — also kills double-tap zoom and pinch), manipulation on fixed chrome, none on drag surfaces such as scrubbing charts and the page gauge. Pair with a gesturestart preventDefault for older iOS pinch — facet.js does this globally. Never rely on viewport user-scalable=no or maximum-scale; iOS ignores both.

App shell law. For a standalone PWA

App shell law. For a standalone PWA, viewport-fit=cover alone is not enough on iOS: without apple-mobile-web-app-capable and apple-mobile-web-app-status-bar-style set to black-translucent, the installed app gets an opaque strip cutting the page at the status bar. Every app page carries both metas, keeps safe-area env() insets in its section padding, and the manifest's background_color and theme_color match the shipped theme.

Caching law. HTML documents and

Caching law. HTML documents and, until versioned freezes exist, the /lib/ files themselves are never cache-first: every refresh loads from the network and caches answer only offline. A worker that serves pages from cache and revalidates behind makes every deploy invisible until a second refresh — that bug shipped once and is now law. facet-sw.js implements this: network-first for pages and /lib/, cache-first with background revalidation for other static assets.

Parallax exclusion. Elements carrying their own transform physics

Parallax exclusion. Elements carrying their own transform physics — velvet lift and press, the pager's rubber-band translate — never register with facetMotion: two writers on one inline transform collide, and the loser flickers.

Font-list law. Never put inherit inside a font-family list (ui-rounded

Font-list law. Never put inherit inside a font-family list (ui-rounded, "SF Pro Rounded", inherit is silently invalid and drops the whole declaration). A stack ends in a generic family, always.

iOS, honestly: what a website can and cannot do

The research pass behind the platform laws, kept current so nobody designs around a capability that does not exist. What a website CAN do on iOS: tint Safari's own chrome with theme-color (facet.js keeps it synced to the active theme's background, live); ship a home-screen icon (apple-touch-icon) and install as a standalone app (manifest + the apple metas from the app shell law); send web push and set an app badge — but only after the visitor adds the app to their home screen (iOS 16.4+), never from a plain Safari tab, and only from a user gesture. What it CANNOT do: see or join Screen Time, read a visitor's age, or apply parental controls — all of that is OS territory, and any design leaning on it is designing fiction.

Child safety, the honest version: there is no web API for it. A site's real levers are self-labeling and restraint — adult content declares itself with the RTA rating meta (<meta name="rating" content="RTA-5042-1996-1400-1577-RTA">) so OS filters like Screen Time's "Limit Adult Websites" catch it; family content can state isFamilyFriendly in its JSON-LD; and everything Facet ships is quiet by design — no autoplay, no dark patterns, nothing engineered to hook anyone, children included.

The head pack every Facet app page carries:

<meta name="viewport" content="width=device-width, initial-scale=1, viewport-fit=cover">
<meta name="theme-color" content="#FFFFFF"> <!-- facet.js keeps this synced to the theme -->
<meta name="apple-mobile-web-app-capable" content="yes">
<meta name="apple-mobile-web-app-status-bar-style" content="black-translucent">

Analytics: bring your own vendor

No analytics tool is baked in. Every key action in the library carries a stable, documented data-event name — the same hooks agents target. This five-line bridge wires any vendor to all of them at once; swap the track call for your tool's API and you are done. 

<script>
  addEventListener("click", (event) => {
    const hook = event.target.closest("[data-event]");
    if (hook) yourAnalytics.track(hook.dataset.event, { page: location.pathname });
  });
</script>
