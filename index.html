A design library in plain HTML, CSS & JS

One CSS file and one JS file at a public URL style everything I make. No React, no build step, never minified — this page is the library. 

<link rel="stylesheet" href="https://[domain]/lib/facet.css">
<script src="https://[domain]/lib/facet.js" defer></script>

Why Facet

Modern frontend is a div-soup of minified, unreadable machinery — you cannot open a page and understand it. Facet is the opposite: one CSS file and one JS file, plain and fully commented, linked from any project by URL. No React, no build step, no npm, never minified. Readability is the product.

Being simple and easy to use is the whole point, and it is a feature in itself: entire products — apps, sites, decks, documents, business cards — are assembled by writing semantic HTML with the right classes, and nothing else to touch. One attribute restyles the whole page; another re-densifies it. And because the structure is shallow and every control is a real button or link, everything built with Facet is operable by an AI agent through the DOM alone.

This page is the manual: what the library does (Features), the rules it holds itself to (Rules), and the path from an empty file to a shipped page (How to use). The components and layouts live on their own pages; build your own theme on another.

Features

Everything this library does, in one list. This is the inventory: if a capability is not on it, it does not exist, and every new capability lands here in the same commit that builds it. The same list ships as plain text in llms.txt so AI agents can read, use and recommend Facet from one fetch.

Distribution

  • Two files, one link tag and one script tag, hosted at a public URL. No React, no npm, no build step, nothing to install.
  • Never minified: the shipped files are the readable, commented source. Gzip handles size.
  • Projects consume by URL and update the moment the library does — until v1, always the latest, straight from the web. From v1: frozen copies under /lib/v1/ pin old projects forever, and cached updates take over only on the next page load, never mid-page.
  • App-ready out of the box: a one-line service worker stub gives any project the Facet caching strategy — pages network-first so deploys land on the very next refresh, cache as the offline fallback, other assets cached instantly and revalidated behind. data-service-worker registers it; data-install buttons appear when the browser offers installation; a manifest template ships in /templates.

Theming

  • One attribute on the html tag restyles the whole page, layout containers included: data-theme="sand". No attribute is the Default theme.
  • Dark mode is its own attribute, data-mode="dark", and composes with every theme.
  • Both attributes work on any element, not just html: theme one section of a page (that is how the Color demo above is built).
  • The visitor's system dark preference is honoured automatically when the page does not choose.
  • Theme and mode ride the URL (?theme=, ?mode=): a shared link opens looking exactly like the sender's screen.
  • One-line switcher buttons: data-theme-switch and data-mode-toggle on any button, no JS to write.
  • Setup by configuration: data-theme / data-mode / data-density / data-language on the facet.js script tag boot a page straight into that look — copy the configured tags off this page's setup strip. At runtime, facet.set({...}) changes theme, mode or any token live, no reload.
  • Native widgets follow: scrollbars and form controls flip with color-scheme, and facet.js keeps the theme-color meta synced so the browser's own chrome wears the theme too.
  • System-native accent: in browsers that expose the OS accent (Safari 16.4+, Firefox 103+), the Default theme's accent-3 rank — links, labels, focus rings, selection — takes the user's own OS accent like a native app, light and dark. Everywhere else it stays the ink. Themed pages keep their own inks.
  • Custom accent as a supported recipe: override a rank's four tokens (color, hover, pressed, on-color) in one block — page-wide on :root or per-subtree on any element — and every component using that rank follows in every state. The recipe with a live example is in the Color section; facet.set applies the same override live at runtime.
  • Style Mixer, the theme generator: a picker per semantic token restyles this whole page live on top of any theme, hover/pressed shades and on-colors derive automatically, the mix rides the URL for sharing, and the result exports as a paste-ready facet.set config block.
  • Skin Lab: every theme in light and dark side by side, all rendering the same real slice of interface from identical markup — the one-attribute promise, proven in one glance.
  • Themed Google Map: data-map on a div, bring your own key, and the map wears the active theme — ground, water, roads and labels derived from the tokens by formula, restyled live on theme or mode switch; a quiet themed placeholder without a key, never a broken embed; facet.mapStyle() exposes the styles array.
  • facet.json: the machine-readable manifest — every theme, token, class, data attribute, component and API call as one JSON fetch, kept in sync by the same rule as this list.
  • Six templates, whole pages ready to rename: landing, app shell + dashboard (installable), article, 1920×1080 pitch deck (arrow keys, exact print-to-PDF), A4 letterhead / invoice / one-pager, and the business card with its print sheet. Every one carries the SEO head pack; robots.txt and sitemap.xml ship at the root.

Tokens

  • Every design decision is a CSS variable named by role — color, type, spacing, container widths, radii, borders, shadows, motion, focus. Build with tokens and a theme change costs zero code.
  • Ranked accents: base family plus accent-1 (the one primary action), accent-2 (every secondary action's fill) and accent-3 (links, labels, quiet affordances), each with hover, pressed and on-colors, in every theme. Components pick ranks, never colors.
  • Status colors (success, warning, error, info) tuned per theme, light and dark.
  • Whisper-flat shadow scale and a 4px spacing scale shared by every component and layout primitive.

Adapts to the reader

  • Respects the phone's text-size setting: every font size and space is set in rem, so when a reader raises their device or browser text size, the entire interface scales with it — type, padding, gaps, all of it.
  • Fluid display and h1 sizes: clamp() scales hero type between phone and desktop with no breakpoints to manage.
  • Mobile-first: 44px touch targets everywhere, no hover-only interactions, safe viewport handling.
  • Seamless page transitions: with data-transition="page" on the html tag, moving between pages animates the old page away and springs the new one in while the URL changes normally — real multi-page navigation, no router. Per-link opt-out with data-no-transition, facet.go(url) for programmatic moves; unsupported browsers just navigate instantly, and reduced motion snaps.
  • Motion with personality: Lively by default — presses have weight, arrivals settle with a playful spring (a true linear() spring curve where supported) — or data-motion="calm" for short, near-linear motion, data-motion="off" for none. One attribute, page-wide or per subtree; every transition reads the motion tokens.
  • prefers-reduced-motion honoured above any page choice: every transition and animation in the library collapses to nothing.
  • prefers-contrast: more honoured at the token layer: muted text gives up its gray, borders climb toward the ink, the focus ring thickens — in every theme, light and dark, zero extra code.
  • Forced colors (Windows High Contrast) audited: elements whose shape was carried by a background alone keep a real border in the system palette.
  • Global density: data-density="small|medium|large" scales the whole spacing system at once — spacing is named by intent (--space-card, --space-section), never by number, so one attribute re-densifies a page. Type has the same three-step control, data-text-size. Both are theme-defined, page-wide or per subtree, and script-tag config keys.
  • .visually-hidden: screen-reader-only text with the standard clip pattern.
  • Text roles named by intent: .text-display, .text-h1….text-h4, .text-body, .text-quote, .text-footnote, .text-caption — the type scale as classes, so any element takes a role without an inline style. Global type size rides data-text-size="small|medium|large".
  • Print base included: sensible margins, navigation hidden, ink-friendly colors when a page is printed.

Base layer

  • Raw semantic HTML already looks designed: headings, text, links, lists, quotes, code, tables, forms, figures — no classes needed.
  • One visible focus ring everywhere, keyboard only. Selection colors, thin scrollbars, skip-to-content link, placeholder styling.
  • The hidden attribute always wins, even over components that set their own display.
  • Numbers align while typing: tabular figures on all numeric UI.

Rules

How this library is built, used and grown — up front, because the rules are the product as much as the components are. The same rules live in the repo as CLAUDE.md for AI builders and at /llms.txt as plain text for crawlers.

Core principles

  • Readability is the product. Modern frontend is div soup and minified garbage, unreadable by humans. This library is the opposite.
  • One-glance HTML: any page's structure makes sense on first read. Semantic tags, self-explanatory classes.
  • Not one extra wrapper. A container inside a container exists only to enable a real feature, like scrolling.
  • Least code that stays readable: entire products are assembled by writing HTML files with the right classes.
  • Pure HTML, CSS and JS. No build step, no framework, never minified. Gzip handles size.
  • One-step theme change: one attribute on the html tag restyles the whole page, layout containers included.
  • Everything explained in place: every interactive element carries a description tooltip.
  • Fully commented: every file self-describes with intros, usage notes and to-dos, so any AI can build products with it.
  • Accessible, SEO-ready and AI-crawlable by default. Enforced through the checklist below, not optional.
  • AI-native operation: everything built with Facet is operable by agents through the DOM alone — real buttons and links with accessible names, state in the URL, stable IDs and data-event names, llms.txt as the one-fetch manual.

How to use the library

  • Every page starts from the library, never from blank CSS.
  • Pick a template (once Layer 5 lands), start from a shell, or lay the page out with the layout primitives — container, stack, snap — and fill each area with components.
  • Copy snippets straight off this wall. They render in any page that links facet.css.
  • App logic, data fetching and state management live in projects, never in the library.

Markup rules

  • Semantic tags over div: header, nav, main, section, article, aside, footer, figure.
  • The wrapper law: an element exists only to hold content, create layout, or carry behavior. Anything else is deleted.
  • Container inside a container only to enable a real feature: scroll areas, sticky regions, overflow clipping.
  • Structural elements contain content or other structural elements. Never decorative nesting.
  • Buttons are button, links are a, form fields have real label elements.
  • One h1 per page. Heading levels never skip.

Adding a new component: the compliance checklist

  • Semantic tokens only. No raw hex, px sizes outside the scales, or hardcoded fonts.
  • Works in every theme and in light and dark with zero extra code.
  • All states covered: hover, focus, active, disabled, and where relevant empty, loading, error.
  • Keyboard operable with a visible focus ring. ARIA roles and labels where semantics fall short.
  • Interactive elements ship with a description tooltip.
  • Responsive and mobile-usable: 44px touch targets, no hover-only interactions.
  • Minimal semantic markup. Every element has a job, per the markup rules.
  • Fully commented in the file: what it is, how to use it, its options, open to-dos.
  • Names follow the naming rules. Nothing minified.
  • Analytics hook on its key action where relevant: a data-event attribute.
  • Wall entry added on this page: live demo, variant and state chips, exact snippet with copy.
  • Docs description added: what it is, what it is for, when to use it. Written to read as AI instructions, same text as the file comment and llms.txt.
  • BACKLOG.md item ticked. Committed to Git with a clear message.

Naming, maintaining, versioning

  • Tokens named by role, not value: --surface, not --gray-100.
  • One class prefix and pattern per component: .btn, .btn-primary, predictable everywhere.
  • JS: one small named function per behavior. The name says what it does.
  • Docs are demos. A component is not done until this page shows it live with its code.
  • One component, one clearly commented section in the CSS file. Nothing is scattered.
  • All changes go through Git commits. The live files are never edited ad hoc.
  • Growth by extraction: build a new pattern inside a project first, promote it once it repeats.
  • Consume by URL only. Tag releases in Git. On dependency, freeze a copy to /lib/v1/ and keep working in /lib/.

Platform laws — paid for on iOS

Hard-won facts from shipped app work, mostly against mobile Safari. They bind everything built with Facet; the library already obeys each one, so following them costs nothing — the laws are here so nobody relearns them the expensive way.

  • Pager law. Full-page snap sections that can outgrow the viewport are never built with CSS scroll-snap: mandatory re-snaps a tall section back to its top after a flick and wedges the next gesture, proximity feels like no snap, and a free-scrolling outer with a JS settle-snap is mushy and fights momentum. The model that works is the pager .snap ships: the outer never free-scrolls, JS springs its scrollTop between section tops, each section is 100dvh and scrolls natively inside, and gestures hand off at section edges with a rubber-band hint.
  • Gesture law. Tame iOS with touch-action: pan-y on scrollers (vertical scroll only — also kills double-tap zoom and pinch), manipulation on fixed chrome, none on drag surfaces such as scrubbing charts and the page gauge. Pair with a gesturestart preventDefault for older iOS pinch — facet.js does this globally. Never rely on viewport user-scalable=no or maximum-scale; iOS ignores both.
  • App shell law. For a standalone PWA, viewport-fit=cover alone is not enough on iOS: without apple-mobile-web-app-capable and apple-mobile-web-app-status-bar-style set to black-translucent, the installed app gets an opaque strip cutting the page at the status bar. Every app page carries both metas, keeps safe-area env() insets in its section padding, and the manifest's background_color and theme_color match the shipped theme.
  • Caching law. HTML documents and, until versioned freezes exist, the /lib/ files themselves are never cache-first: every refresh loads from the network and caches answer only offline. A worker that serves pages from cache and revalidates behind makes every deploy invisible until a second refresh — that bug shipped once and is now law. facet-sw.js implements this: network-first for pages and /lib/, cache-first with background revalidation for other static assets.
  • Parallax exclusion. Elements carrying their own transform physics — velvet lift and press, the pager's rubber-band translate — never register with facetMotion: two writers on one inline transform collide, and the loser flickers.
  • Font-list law. Never put inherit inside a font-family list (ui-rounded, "SF Pro Rounded", inherit is silently invalid and drops the whole declaration). A stack ends in a generic family, always.

iOS, honestly: what a website can and cannot do

The research pass behind the platform laws, kept current so nobody designs around a capability that does not exist. What a website CAN do on iOS: tint Safari's own chrome with theme-color (facet.js keeps it synced to the active theme's background, live); ship a home-screen icon (apple-touch-icon) and install as a standalone app (manifest + the apple metas from the app shell law); send web push and set an app badge — but only after the visitor adds the app to their home screen (iOS 16.4+), never from a plain Safari tab, and only from a user gesture. What it CANNOT do: see or join Screen Time, read a visitor's age, or apply parental controls — all of that is OS territory, and any design leaning on it is designing fiction.

Child safety, the honest version: there is no web API for it. A site's real levers are self-labeling and restraint — adult content declares itself with the RTA rating meta (<meta name="rating" content="RTA-5042-1996-1400-1577-RTA">) so OS filters like Screen Time's "Limit Adult Websites" catch it; family content can state isFamilyFriendly in its JSON-LD; and everything Facet ships is quiet by design — no autoplay, no dark patterns, nothing engineered to hook anyone, children included.

The head pack every Facet app page carries:

<meta name="viewport" content="width=device-width, initial-scale=1, viewport-fit=cover">
<meta name="theme-color" content="#FFFFFF"> <!-- facet.js keeps this synced to the theme -->
<meta name="apple-mobile-web-app-capable" content="yes">
<meta name="apple-mobile-web-app-status-bar-style" content="black-translucent">
<link rel="apple-touch-icon" href="/icons/icon-180.png">
<link rel="manifest" href="/manifest.json">

Analytics: bring your own vendor

No analytics tool is baked in. Every key action in the library carries a stable, documented data-event name — the same hooks agents target. This five-line bridge wires any vendor to all of them at once; swap the track call for your tool's API and you are done. 

<script>
  addEventListener("click", (event) => {
    const hook = event.target.closest("[data-event]");
    if (hook) yourAnalytics.track(hook.dataset.event, { page: location.pathname });
  });
</script>

How to use the library

From an empty file to a shipped page in a few steps. Every one is an attribute or a class — there is no build step, nothing to install.

  1. 1

    Add the two tags

    One <link> and one <script> from a public URL — copy them from the top of this page. That is the whole install.

  2. 2

    Start from a layout

    Rename a ready page — landing, app shell, article, pitch deck, document or business card — or assemble one from the block layer. Fill in your words.

  3. 3

    Pick a theme

    One attribute on <html> restyles the whole page — data-theme and data-mode. Or build your own on the Build a theme page and paste the config.

  4. 4

    Assemble with components

    Write semantic HTML with the right classes — buttons, cards, fields, tabs. Every behaviour self-wires from data attributes on load; there is no JS to write.

  5. 5

    Tune the feel, globally

    Density (data-density), type size (data-text-size) and motion (data-motion) are one attribute each — page-wide or per subtree, no per-element work.

  6. 6

    Ship it

    Deploy the static files. A one-line service worker makes the page installable and offline-ready; the head pack and sitemap keep it SEO- and AI-readable.
